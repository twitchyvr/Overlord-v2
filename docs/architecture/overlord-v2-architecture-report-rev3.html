Core Principle

Don't Change the Agent — Change the Framework

The single architectural decision that makes everything else possible.

Rooms define behavior, not agents. When an agent enters a room, the room's rules, tools, and output templates merge into their context. Change the testing room rules once — every agent that enters inherits them. Add a new agent? It just works in existing rooms.

v1 Agent-Centric

Each agent carries 200+ line system prompts with behavioral rules. 15 agents = 15 places to maintain rules. Change one, miss the others. 4-tier approval system with confidence scores and broken tier registry.

v2 Framework-Centric

Agents are 10-line identity cards (name, capabilities, room access). Rooms own the rules. One change propagates to every agent that enters. Tool access is structural, not instructional.

v2+ Scriptable

Rooms, floors, tables are plugins — scriptable modules (Lua or JS). Even built-in features are implemented as plugins. Users can create custom room types for use cases we haven't imagined.

Onboarding

Phase Zero: The Strategist

Before discovery. Before anything. A consultative setup that builds your building for you.

New User (Empty Building)

  • 1. "What are you trying to build/do?"
  • 2. "What does success look like?"
  • 3. "What inputs do you have? What outputs do you need?"

Audit Trail

Exit Documents & RAID Log

"You can't just say 'done.' You have to show you actually thought about it." — the risk questionnaire principle.

✘ v1: "It says it's done"

  • Agent says "Done!" with no structured proof
  • Tests pass because there are no tests for missing features
  • UI looks complete but buttons aren't wired up
  • No audit trail of decisions made during execution
  • No way to reference what was decided in earlier phases
  • Scope changes lose all context

✓ v2: Prescriptive Exit Protocol

  • Agent must complete structured exit document to leave room
  • Exit doc requires: decisions, rationale, evidence, citations
  • RAID log entry: Risks, Assumptions, Issues, Decisions
  • Every phase's exit doc becomes next phase's input
  • Searchable RAID log — agents reference past decisions
  • Scope changes trigger re-entry with full history

RAID Log Schema

// Searchable database: every agent references this before starting work { "id": "raid_001", "type": "decision", // risk | assumption | issue | decision "phase": "architecture", "roomId": "arch-room-1", "summary": "Use SQLite for MVP, migrate to Postgres later", "rationale": "Simplicity for phase 1, single-file deploy...", "decidedBy": "architect-agent", "approvedBy": "user", "affectedAreas": ["storage", "deployment"], "timestamp": "2026-03-10T...", "status": "active" // active | superseded | closed }
Change Management

Mid-Stream Scope Changes

"What happens if we're in test phase but want to add a new feature?" — structured re-entry.

Scenario: Project is in TESTING phase. User wants to add a new feature. User: "I want to add search filtering before we deploy" | Orchestrator checks: 1. Current phase = Testing 2. Requested work = New feature (requires Discovery + Architecture + Execution) 3. Scope change detected | Scope Change Protocol: a. Query RAID log for all decisions made so far b. Package relevant decisions as context brief c. Re-enter Discovery Room with context brief attached d. Discovery room knows: "We are in test phase, adding scope" e. Discovery output = mini-requirements (scoped to new feature only) | f. Fast-track through Architecture (existing architecture + delta) g. Fast-track through Execution (scoped to new files only) h. Return to Testing (re-run all tests including new feature) | Result: New feature integrated without losing any previous context. All downstream dependencies updated via RAID log cascading.
Key principle: When scope changes, agents don't just "go build." They re-enter the relevant rooms with the full RAID log context brief. This prevents the AI from ignoring previous work.

Agent System

Agents: Simple Identity, Room-Driven Behavior

v1 200-line prompts per agent

{ name: "testing-engineer", instructions: ` You are a testing engineer. Lessons Learned

What v1 Taught Us

▶

Bug API Error 2013: Orphaned tool_results

Cause: Tool denial paths injected bare assistant messages, breaking Anthropic's strict alternation rule. _sanitizeAlternation() in chat-stream.js was a band-aid. v2 fix: Append-only message history validated at insert time. Impossible to create orphans by construction. Messages stored per-room, not globally.

▶

Bug Tool loop stops after one cycle

Cause: Three simultaneous root causes: (1) addToolResult not registered as a service method, (2) wrong continueLoop condition checking for tool_calls on the response instead of results, (3) toAnthropicMessages() not converting internal format to Anthropic API format. v2 fix: Strict layered architecture prevents these cross-cutting bugs. Tool results flow through a typed pipeline, not ad-hoc method calls.

▶

Bug 25+ tools missing from tier registry

Cause: approval-flow.js had a hardcoded tier map that wasn't updated when new tools were added. Unregistered tools defaulted to "unknown" tier, prompting for approval on every read_file call. v2 fix: No tier registry at all. Room-scoped access is binary: tool exists in room or it doesn't.

▶

Pattern "It says done but nothing works"

Cause: No structured proof of work. Agent says "Done!" without evidence. v2 fix: Exit documents require: files modified, tests passed, decisions made, risk assessment. The risk questionnaire pattern — agents must fill in structured fields with evidence, not just say "approved."

▶

Pattern Testing agent modifies tests to pass

Cause: Agent prompt said "don't change tests" but write_file was in its tool list. Instructions are suggestions to AI; tool access is structural. v2 fix: Testing Lab room's tool list doesn't include write_file. Structurally impossible, not instructionally prohibited.

▶

Pattern Circular dependencies everywhere

Cause: Hub-spoke with no layering. orchestration-core requires agent-session which requires orchestration-core (lazy require() hacks). 61 shared state getters in orchestration-state.js accessed by 8+ modules. v2 fix: Strict layer ordering: Transport → Rooms → Agents → Tools → AI → Storage. Each layer only depends on layers below.

▶

Pattern MiniMax-specific hacks in 6+ files

Cause: MiniMax corrupts Unicode, requires emoji stripping, has different temperature range, sends tool calls in text format sometimes. Workarounds scattered across ai-client.js, chat-stream.js, message-builder.js, guardrail-module.js, character-normalization.js, tool-parser.js. v2 fix: One adapter per provider. All quirks contained in the adapter file. Rest of system speaks one internal format.

▶

Insight Scope changes lose all context

Cause: No structured record of decisions. No RAID log. When scope changed mid-project, agents started fresh with no awareness of prior work. v2 fix: Every phase produces an exit document. Scope changes trigger re-entry to relevant rooms with a context brief summarizing all RAID log decisions.

▶

Insight No phase gates = premature execution

Cause: AI starts building immediately with no discovery phase. No requirements, no architecture review. v2 fix: Phase Zero (Strategist) asks consultative questions first. Phase gates require structured sign-off. Can't enter Execution without Architecture sign-off.

▶

Insight 137 socket handlers in one file

Cause: hub.js grew organically. Every new feature added more handlers. At 137 handlers and 2300+ lines, it's the single largest bottleneck for understanding, modifying, or debugging the system. v2 fix: Transport layer with typed event contracts. Socket handlers organized by domain (room events, agent events, system events). Each domain module owns its handlers.
